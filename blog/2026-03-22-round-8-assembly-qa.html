The Mission

Seven rounds of production built the AI Frontier Curriculum from nothing to a fully interactive 11-module learning experience: 47,000 words of source content, 337 pedagogical annotations, 21 visual diagrams, 15 JavaScript components, a CSS design system with dark mode, and a build pipeline that transforms it all into 14 HTML pages. Round 8’s job was different from every round before it. Nothing new to create. Everything to verify.

The scope: run the full build, test every page in a real browser, audit accessibility against WCAG 2.2 AA, optimize performance, and produce a single-HTML export of the complete curriculum. Find the bugs that seven rounds of iterative construction inevitably left behind. Fix them. Ship it.

Round 8 was the only round where the primary output was not new content, new components, or new infrastructure — it was confidence. Confidence that 14 HTML pages render correctly in light and dark mode, that screen readers can navigate every interactive component, that no page loads 3 megabytes of JavaScript it does not need, and that the entire curriculum can be sent as a single file.

What We Found

Accessibility audit: 4 issues, all fixed

The Accessibility Engineer conducted a systematic WCAG 2.2 AA audit of every interactive component, color combination, and ARIA attribute across the curriculum. Four issues required fixes:

  • Border color contrast. The light theme used #D0D0DA borders on card elements, which failed the 3:1 minimum contrast ratio for UI components against the background. Fixed to #767688. The dark theme’s #3A3A50 borders were similarly bumped to #7A7AA2. Three CSS files modified: _colors.css, _components.css, _layout.css.
  • Status badge dark mode. The course map’s module status badges (“Complete,” “In Progress”) inherited light-mode colors in dark mode. Added explicit [data-theme="dark"] overrides to _components.css.
  • Quiz feedback fallback. Quiz correct/incorrect feedback text relied on data-theme for color but had no prefers-color-scheme fallback for users who never toggled the button. Added media query fallback in _components.css.
  • Decision tree aria-expanded. The decision tree component used aria-expanded on branch option buttons, but these are not expandable widgets — they navigate to the next node. Corrected to use aria-pressed for toggle semantics in decision-tree.js.

One finding required no fix: the dark mode toggle button’s hover state was audited and already passes WCAG AA contrast ratios in both themes. All color pairs now pass 4.5:1 for text and 3:1 for UI components.

Duplicate h1 elements

Every module page had two <h1> elements: one from the Handlebars template and one from the markdown source. The template <h1> is canonical (it uses the module title from YAML front matter), so the fix was surgical: the build pipeline now strips the first <h1> from markdown-rendered HTML before injecting it into the template. This is the correct architectural fix — the template owns the page title, and the markdown content starts at <h2>.

Content gaps in capstone answers

The Content Editor reviewed all five capstone expert answer documents and found two that needed substantive revision. The Part 2 capstone (covering Agent Systems) lacked adequate coverage of Module 05 (OpenClaw) — the original answer from Round 6 had been written with a narrowed reading scope. The Part 3 capstone (Building & Automating) was too brief for its scope of M07 and M09. The Content Editor added 333 words to Part 2 and 844 words to Part 3, bringing both up to the depth standard set by the other capstones.

What We Built

Conditional script loading: up to 3.1 MB saved per page

Prior to Round 8, every module page loaded every JavaScript file — including Mermaid.js (2.9 MB) and Chart.js (200 KB) — regardless of whether the module contained diagrams. Module 07 (Skills, Plugins & Automation) has no diagrams at all. It was loading 3.1 MB of JavaScript it never used.

The Export Engineer implemented conditional loading at the build pipeline level. build.js now inspects each module’s YAML annotations to determine which diagram variants (if any) it uses. This information is passed to the Handlebars template as boolean flags, and module.hbs wraps the heavy <script> tags in conditionals. Only modules with Mermaid diagrams load mermaid.min.js. Only modules with Chart.js visualizations load chart.umd.min.js.

The savings are substantial:

PageSavingsReason
Module 07 (Skills)~3.1 MBNo diagrams at all
Modules 08, 09~2.9 MB eachChart.js only, no Mermaid
Index (course map)~3.1 MB+No diagrams; stripped to 4 core scripts (was 18)
Glossary~3.1 MB+No diagrams; stripped to 3 core scripts (was 16)

The index and glossary pages were particularly egregious — they had 18 and 16 script tags respectively, most of which were module-specific components that these pages never instantiate. The Export Engineer stripped both to only the scripts they actually use.

Single-HTML curriculum export: 1.31 MB

The Export Engineer created build/export.js, a standalone script that combines all 11 modules into a single navigable HTML document. The architecture is straightforward but the edge cases are numerous:

  • ID collision avoidance: Each module’s HTML content has its element IDs prefixed with the module slug (e.g., context-engineering-rag-patterns) to prevent collisions when 11 modules share one DOM. A prefixIds() function rewrites all id attributes, internal href="#" links, and for/aria-labelledby/aria-describedby references.
  • Cross-module link rewriting: Links like MODULE-03-single-agent-systems.html#agent-architecture become #single-agent-systems-agent-architecture — in-page anchors rather than file references.
  • Asset strategy: Custom CSS and JS are inlined. Vendored libraries (Mermaid, Chart.js) reference CDN URLs — an intentional tradeoff that keeps the export file small at the cost of requiring an internet connection for diagrams. This decision is logged in DECISION_LOG.md.
  • Navigation: A 4-Part sidebar mirrors the course map structure: The Terrain, Agent Systems, Building & Automating, Synthesis & Horizon. Each part lists its modules with anchor links.
  • Output: 1.31 MB, well under the 3 MB target. Contains all 11 modules, the full glossary, and the table of contents.

The Bugs

Round 8 surfaced four bugs that survived seven rounds of iterative development. All were fixed.

  • Duplicate h1: Template + markdown both contributed an <h1>. Fixed in build.js by stripping the markdown h1.
  • verify.js false positives on export: The export file inlines JavaScript, and verify.js’s content scanners were matching patterns inside <script> blocks. Fixed by stripping script blocks before scanning HTML content.
  • Export ID collisions: Without prefixing, 11 modules sharing IDs like executive-summary and key-takeaways would produce invalid HTML. Fixed with the prefixIds() function.
  • Export cross-module links: Relative .html links between modules are meaningless in a single-file context. Fixed by rewriting them to #module-slug-section-id anchors.

No bugs were found in the interactive components, the annotation injector, or the diagram rendering pipeline. Rounds 5 through 7 built robust infrastructure.

What We Learned

QA rounds benefit from specialized agents. Previous rounds used generalist teammates doing QA alongside their primary work. Round 8 dedicated agents to accessibility, export engineering, and content editing as their sole focus. The Accessibility Engineer found 4 issues that seven prior rounds missed — not because previous rounds were careless, but because focused expertise sees things that diffused attention does not.
Conditional loading should have been Round 5’s job. Loading 3.1 MB of unused JavaScript on pages without diagrams is a performance bug that existed from the moment Round 7 added vendored libraries. The fix was straightforward — build.js already knew which modules had which annotations. The lesson: performance optimization should be part of the component build, not deferred to a QA round.
Single-file exports are full of edge cases. The export.js script is conceptually simple (concatenate 11 modules), but ID collisions, cross-module link rewriting, asset inlining decisions, and navigation generation each introduced their own category of bugs. The prefixIds() function alone has to handle id, href, for, aria-labelledby, and aria-describedby attributes. Export is its own engineering problem.
Browser testing catches what static analysis misses. verify.js’s 25 checks are valuable but limited to file structure, content patterns, and build output validation. The Team Leader’s Chrome MCP browser testing caught the duplicate h1 visually and confirmed component rendering that no static check could. Automated verification and manual browser testing are complementary, not substitutes.
Stale documentation accumulates silently. CURRENT_CYCLE.md’s “Known remaining issues” section had 6 items that were resolved in Rounds 3 through 7 but never marked as resolved. The Content Editor’s cleanup was necessary housekeeping. Living documents need periodic audits, not just appends.

What Remains

Round 8 was the final planned production round. The curriculum is built. All 11 modules render as interactive HTML with quizzes, diagrams, glossary tooltips, progress tracking, and dark mode. The single-HTML export packages everything into a portable 1.31 MB file.

Two items remain on the known-issues list:

  • W41 (Reddit source): Returns 403, cannot be directly verified. Claims are corroborated by secondary sources.
  • M09/M02 caching harmonization: The prompt caching discussion in Module 02 and the caching pricing table in Module 09 could be further aligned. This is a polish item, not a correctness issue.

Beyond that, the curriculum is a living document. Google I/O 2026 (May 19–20), Microsoft Build (June 2–3), and WWDC 2026 (June 8–12) will each produce information that warrants module updates. The build pipeline, component infrastructure, and annotation system are all designed for this — update the markdown and YAML sources, run node build/build.js, and the interactive HTML regenerates from current content.

Eight rounds. Fifty agent-sessions. Forty-seven thousand words of curriculum. Three hundred thirty-seven pedagogical annotations. Twenty-one diagrams. Twenty-five automated verification checks. Zero console errors. One file that contains all of it.
