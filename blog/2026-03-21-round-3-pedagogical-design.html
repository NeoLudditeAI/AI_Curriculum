The Mission

Rounds 1 and 2 built the content: 47,169 words across 11 modules, 83 verified sources, every factual claim sourced. The curriculum existed as dense, accurate reference material. But reference material is not a learning experience. The question driving Round 3 was: how do you transform a knowledge base into something that actually teaches?

The answer came from learning science. A deep research pass produced an 8-section evidence base covering learning pathway design, retrieval practice, engagement, visual elements, spaced repetition, instructional design frameworks, assessment, and accessibility. Four frameworks emerged as the design foundation: Merrill’s First Principles (problem-centered module structure), van Merriënboer’s 4C/ID (worked-example fading across the curriculum), Backward Design (learning outcomes drive everything), and Cognitive Load Theory (component design and progressive disclosure).

A key decision preceded the design work: the audience broadened from Ryan alone to his social circle — friends and colleagues he mentors on AI topics. This justified the full pedagogical investment. A curriculum for one reader does not need scaffolded quizzes or expertise-reversal management. A curriculum for a small cohort of varying expertise levels does.

Round 3 deployed a 10-person Opus 4.6 team with specialized pedagogical roles — a sharp departure from Rounds 1 and 2, which used content researchers, writers, and reviewers. This team had no content creators. Every agent was a designer.

Phase 1: Foundations (WS0, WS1)

Phase 2a used Agent Teams (TeamCreate) with 4 persistent teammates: Learning Architect, Accessibility Lead, Visual Strategist, and Citation Standardizer. The key advantage was inter-agent messaging — when the Accessibility Lead completed WS0’s structural draft, the Visual Strategist received the handoff directly and added component behavior specifications. This conversational coordination is something background agents cannot replicate.

Two foundation workstreams ran in parallel: WS0 (component library) and WS1 (course architecture). Every downstream workstream depended on one or both.

WS0: Component library

The Accessibility Lead drafted the structural specifications for 9 reusable components: quiz widget (3 variants), progressive-disclosure accordion (3 variants), comparison-table enhancer, glossary tooltip, concept-check gate, worked-example fader, self-explanation prompt, progress tracker, and navigation controls. Each component was specified with semantic HTML, ARIA roles, keyboard navigation, and screen reader behavior.

A critical architectural decision: the build pipeline. Canonical markdown is never modified. Pedagogical elements are defined in separate YAML annotation files and injected during a Node.js build step (unified + remark). Content authors edit markdown. Pedagogical designers edit annotations. The build merges them. This separation of concerns means Round 4 can implement the build system without touching any Round 1-2 content.

After the structural draft, the Visual Strategist added component behavior specifications: state machines, event catalogs, animation durations, error states, touch device considerations, and a CSS class architecture using BEM-like afc- prefixes.

WS1: Course architecture

The Learning Architect read all 11 modules and determined that the numerical order (M00-M10) was the authoring order, not the optimal learning order. Three modules moved: M09 (APIs) shifted later because API design choices only make sense after understanding agent tool use. M05 (OpenClaw) moved after M03 and M06 because it references agent architecture and MCP throughout. M08 (Consumer Comparison) became the penultimate module, transforming from a standalone comparison into a synthesis exercise.

The 11 modules were grouped into four thematic Parts: The Terrain (M00-M02), Agent Systems (M03, M06, M04, M05), Building & Automating (M07, M09), and Synthesis & Horizon (M08, M10). Total estimated study time: approximately 6 hours across the four Parts, with recommended spacing of at least one day between Parts.

Seven spiral concepts were identified — ideas that recur across multiple modules at increasing depth. Each recurrence point was flagged for explicit callback annotations in WS2a/WS2b, turning implicit connections into explicit learning reinforcement.

Phase 2: Module-Level Design (WS2a, WS2b)

Two Module Designers worked in parallel: one on M00-M05 (12,684 words), one on M06-M10 (12,397 words). Each module received: Bloom’s-taxonomy-tagged learning outcomes, an opening hook (problem scenario per Merrill’s First Principles), prior knowledge activation questions, and section-by-section annotations specifying exactly where each quiz, accordion, worked example, and self-explanation prompt belongs.

A key pattern emerged: component density follows a rhythm. A concept-check quiz every ~800-1,000 words of prose. A section-review quiz at each H2 boundary. One self-explanation prompt per module in the highest-complexity section. A module-assessment quiz before Key Takeaways. This rhythm creates natural cognitive breaks every 4-5 minutes without explicit “take a break” instructions.

Worked-example fading progresses across modules, not just within them. M00 uses full worked examples. M02-M03 introduce partial/completion examples. M04-M05 shift toward guided and independent problems. By Part 4, the learner is expected to synthesize independently. This follows the 4C/ID trajectory.

Phase 3: Cross-Cutting Systems (WS3–WS7)

Phase 2b and 2c hit the flat-roster constraint of Agent Teams: adding 6 more specialists to the existing team was impractical. The solution was to switch to parallel background agents. Wave 1 launched 3 agents (Module Designer A, Module Designer B, Assessment Designer) and Wave 2 launched 3 more (Visual Strategist, Engagement Specialist, Accessibility Lead for WS6). Each background agent received the completed WS0 and WS1 outputs as input.

The Assessment Designer was originally planned for a two-wave split (framework in Wave 1, sample questions in Wave 2). Instead, the full WS3 scope was assigned in a single pass — the agent produced all 5 sections including ~130 sample questions across all modules. Splitting would have required re-reading every module twice.

One hiccup: the Engagement Specialist hit an auth error on first launch and a timeout on retry. Third attempt succeeded. This was the only operational issue across all 10 agents.

WS3: Assessment system

The Assessment Designer specified four quiz types at four granularities: concept checks (1-2 questions, 30-60 seconds, after each key concept), section reviews (3-5 questions, 3-5 minutes, end of each H2), module assessments (8-12 questions, 10-15 minutes, end of module), and cross-module integration quizzes (5-8 questions, drawing on prior modules). All quizzes provide immediate explanatory feedback. No grades, no leaderboards, no pass/fail gates.

WS4: Visual and interactive specs

Forty diagrams were specified across all 11 modules — 21 high priority, 15 medium, 4 low. Each entry includes type, content, labels, interaction design, accessibility requirements, and the specific learning outcome it supports. Six cross-module visual patterns were standardized: decision trees (6 instances), comparison table enhancers (15+ tables), architecture diagrams (8 instances), timelines (4 instances), multi-panel comparisons (6 instances), and callout boxes (all modules). A unified visual grammar specifies shapes, arrows, and colors so learners never need to re-decode visual conventions between modules.

WS5: Engagement design

Twenty mid-section curiosity prompts were written — surprising statistics, open questions, and contrast statements placed at H2 section transitions to maintain engagement. Eleven decision scenarios (one per module) anchor abstract concepts in plausible situations. Four capstone scenarios (one per Part) require synthesizing all prior modules in that Part. Progress visibility operates at four levels: curriculum-wide, part, module, and section.

A critical constraint: engagement serves learning, not entertainment. No points, badges, streaks, or celebratory language. Motivation comes from knowledge gaps and real-world relevance. An anti-pattern registry explicitly bans 10 common design mistakes including seductive details, generic gamification, and quiz-without-feedback.

WS6: Layout and accessibility

The Accessibility Lead specified a complete design system: Inter typeface, 1.25 Major Third type scale, 72-character measure for body text, WCAG 2.2 AA color contrast across a full light/dark palette. Three-column desktop layout (sticky sidebar TOC + main content + breathing room), responsive down to mobile. Every component’s color signal is paired with a non-color indicator.

WS7: Spaced repetition and integration

The Learning Architect designed a cross-module review system with spacing schedules, integration exercises, and cumulative quiz design. The seven spiral concepts from WS1 each have a review schedule: first encounter, reinforcement in a subsequent module, and integration in a Part-boundary exercise. Recommended pacing: one Part per study session, at least one day between Parts.

Assembly and Review

All 9 workstream outputs — 7,711 total lines across the workstream files, approximately 81,400 words of design specifications — were assembled into reference/research/PEDAGOGICAL-DESIGN.md, a master reference document with a table of contents linking to each workstream file, cross-workstream integration notes, a dependency graph, consistency checks, and a Round 4 handoff checklist. The assembly verified that all 11 modules have learning outcomes, all outcomes are testable, all diagrams map to outcomes, and all components follow the shared visual grammar.

Section 9 of the design document aggregates content requests from all workstreams — items that Round 4 must create. Priority 1 items include the 11 YAML annotation files, 40 diagram source files, a synthesized scheduled-tasks comparison table for M07, and a capstone expert answer document. Priority 2 items are content enhancements (curiosity prompts, spiral callbacks, opening hooks) whose text is already written in the workstream files and needs only to be placed via annotations.

Decisions Made

Eight entries were added to DECISION_LOG.md during Round 3:

  1. Audience broadened to social circle: Designing for a cohort justified the pedagogical investment. A curriculum for one reader does not need quizzes. A curriculum for a small group with varying expertise does.
  2. Static HTML + JS delivery format: No server, no login, no database. localStorage for persistence. Portable, archivable, works offline.
  3. Learning sequence reordered: M09 after agents, M05 after M03/M06, M08 penultimate. Conceptual prerequisites extend beyond the dependency graph.
  4. Annotations in separate YAML files: Canonical markdown never modified by the build. Content and pedagogy evolve independently.
  5. Four pedagogical frameworks combined: Merrill for module structure, 4C/ID for cross-module fading, Backward Design for outcome alignment, CLT for component design.
  6. Citation standardization applied: [F#] + [#] convention applied to M00, M07, M10 (deferred from Round 2).
  7. Agent Teams adopted then adapted: TeamCreate for Phase 2a (inter-agent WS0 handoff), background agents for Phases 2b-2c (flat-roster constraint).
  8. Assessment Designer given full scope in one pass: All 5 WS3 sections including ~130 sample questions, eliminating a redundant two-wave split.

Key Discoveries

  • The numerical order was wrong for learning. The CURRICULUM.md dependency graph captures technical dependencies but not conceptual prerequisites. M09 (APIs) is technically independent of M03 (agents), but API design decisions are meaningless without understanding what agents do with those APIs.
  • 81,400 words of design for 47,169 words of content. The pedagogical design document is 1.7x larger than the content it annotates. This was not planned — it reflects the density of design decisions required per content section.
  • Table density dominates. The 11 modules contain 620+ table rows. Comparison tables are the dominant content pattern. This drove the comparison-table enhancer to become the most-used interactive component.
  • Progressive disclosure is essential, not optional. M03 describes 10 agents sequentially. M07 covers 5 skills platforms plus 3 automation platforms. Without expandable sections, these modules are walls of text. Four modules were flagged as “unreadable without progressive disclosure.”
  • Worked-example fading works across modules. The 4C/ID framework suggests fading worked examples from full to partial to independent. Applying this across the four-part structure (not just within modules) creates a curriculum-wide progression from guided to independent learning.

What We Learned

Pedagogical specialists produce fundamentally different output than generalists. Rounds 1-2 used general-purpose researcher/writer/reviewer agents. Round 3 used agents with specific pedagogical roles. The difference is not quality but kind: a Learning Architect thinks about spiral sequencing. A Module Designer thinks about where to place a concept-check quiz. A general-purpose writer would not ask these questions.
Evidence-based design prevents endless bike-shedding. Every design choice in Round 3 cites evidence from the pedagogy research. “Should we add gamification?” — no, evidence shows extrinsic rewards can undermine intrinsic motivation for this audience. “Should quizzes gate progress?” — no, advisory gates respect self-directed learning. The evidence base resolves debates that could otherwise consume entire sessions.
Accessibility is architecture, not decoration. The Accessibility Lead worked on WS0 (the first workstream) and WS6 (a later workstream). Every component was designed with ARIA roles, keyboard navigation, and screen reader behavior from the start. Retrofitting accessibility onto finished components would have required redesigning many of them.
Separation of content and pedagogy is the right abstraction. The YAML annotation file pattern means pedagogical design can change without touching content, and content can be updated without breaking pedagogy. When Module 02 gets updated after Google I/O, its annotation file may need section ID adjustments but no pedagogical redesign.
Design documents can exceed the content they annotate. At 7,711 lines and ~81,400 words, the pedagogical design is 1.7x the curriculum content. This was surprising but reflects reality: specifying how to teach 47K words of dense technical material requires more words than the material itself. The ratio would decrease for less dense content.
Agent Teams and background agents serve different coordination patterns. Agent Teams (TeamCreate) excels at interdependent work where agents need to build on each other’s output conversationally — the WS0 structural-draft-to-behavioral-spec handoff was a perfect use case. Background agents are better for embarrassingly parallel workstreams where each agent works independently from shared inputs. The flat-roster constraint of Agent Teams forced the hybrid approach, but it turned out to be the right architecture anyway: use teams for foundations, parallel agents for independent elaboration.
