The Vision

The goal was ambitious: build a comprehensive, continuously-updated curriculum covering the bleeding edge of LLMs, consumer AI tools, agentic systems, and the surrounding ecosystem — all as of March 2026. Not surface-level overviews, but the kind of dense, structured expertise that lets you make architectural decisions, evaluate new tools, and teach others.

Eleven modules. Four major AI ecosystems (Anthropic, OpenAI, Google, Microsoft) plus specialized tools and the open-source world. Context engineering, single agents, multi-agent orchestration, MCP, skills and automation, consumer comparison, developer APIs, and frontier topics. Every non-obvious claim sourced. Every module reviewed.

The catch: we wanted to build the entire thing in a single session, using Claude Code's multi-agent capabilities to maximize parallelism. This blog post documents how that went — the wins, the failures, and what we'd do differently.

The Architecture

Before writing a single word of curriculum content, we built a complete project infrastructure:

  • AGENT-INIT.md — operational directives mapping software engineering concepts (tests, builds, deploys) to curriculum equivalents (fact verification, derivative generation, publication)
  • Four specialized agent types defined in .claude/agents/: researcher (Haiku, fast web research), writer (Sonnet, module drafting), reviewer (Sonnet, fact-checking), synthesizer (Opus, cross-module analysis)
  • Path-scoped rules in .claude/rules/ enforcing module structure, sourcing standards, and repository conventions
  • Five living documentation files: CURRICULUM.md (module status), CURRENT_CYCLE.md (session work order), DECISION_LOG.md (structural decisions), SOURCES.md (source registry), GLOSSARY.md (canonical definitions)

The module dependency graph dictated our execution order:

Modules 00-01 (foundation) → Modules 02, 03, 05, 06, 09 (core, parallel) → Modules 04, 07, 08 (dependent, parallel) → Module 10 (capstone) → Synthesis pass

This graph was designed for maximum parallelism: the core batch has five modules with no cross-dependencies, meaning five writers could work simultaneously.

Batch 1: Foundation Modules (00 + 01)

Step 1: Research (8 agents in parallel)

We launched 8 researcher agents simultaneously — 4 for Module 00 (Landscape Overview) and 4 for Module 01 (Models & Intelligence Tiers). Each researcher had a specific brief: market structure, product ecosystem mapping, AI coding tools, open-source landscape for Module 00; Anthropic/OpenAI models, Google/Microsoft models, reasoning modes, multimodal capabilities for Module 01.

All 8 completed within ~90 seconds. The research outputs were rich — detailed pricing tables, benchmark scores, market share data, sourced claims with URLs. This set the pattern for all subsequent batches.

Step 2: Writing (2 agents in parallel)

Module 00 was drafted by a background agent (~4,800 words). Module 01 was drafted by a team member on the newly created curriculum-writers Agent Team (~5,400 words). This was our first use of Agent Teams — the shared task list and direct inter-agent messaging proved valuable for coordination.

Step 3: Review — the first surprise

The reviewer returned NEEDS REVISION for both modules. Module 00 had three blocking issues:

  1. Market share methodology not disclosed. The module cited "68%" for ChatGPT's market share without clarifying this was web traffic share (Similarweb), while the OpenAI profile said "under 45%" — which was mobile app share (Apptopia). Both numbers were real but measured different things.
  2. Sources cited as "researcher agent output" with no primary URLs. This violated our sourcing standard — every claim needs a traceable URL.
  3. Claude Code revenue was stale. The $1B ARR figure was historically correct but understated the current ~$2.5B run rate.

Module 01 had two blocking issues: same source traceability problem, plus "Deep Think 3.1" needed to be renamed to "Gemini 3 Deep Think" per Google's official naming.

Step 4: Revision + Re-review

We spawned reviser agents for both modules. They made surgical fixes. The reviewer re-reviewed and returned PASS for both. Total time for the foundation: research, draft, review, revise, re-review, commit.

Lesson learned: The source traceability issue was systemic. Researcher agents returned detailed data but their output was cited as a single "researcher agent output" source. We fixed this by requiring all subsequent writers to cite primary URLs directly — no more intermediary source citations. This fix propagated cleanly through Batches 2-4.

Batch 2: Core Modules (02, 03, 05, 06, 09)

This was the most ambitious batch: five modules in parallel, each with 3-4 researchers feeding into a dedicated writer. We launched 16 researchers simultaneously across all five modules.

The Stalled Writer Incident

After research completed, we spawned 5 writer teammates on the Agent Team. Four of the five went idle without producing files — all at exactly the same timestamp (03:53:23). Only Module 09's writer completed successfully.

Investigation revealed the writers had actually completed their work — the files existed on disk. The idle notifications arrived before the completion messages reached the lead agent. When we sent status checks, one writer rejected our shutdown request with: "The module file WAS created successfully. Verified just now."

We had spawned duplicate background writers unnecessarily, but no harm was done — the duplicates discovered the files already existed and reported status without overwriting anything.

Lesson learned: Agent Team members can complete work and go idle before their completion message reaches the lead. Check for file existence before assuming a writer failed. The Glob tool is the ground truth, not the message queue.

Review: All Five Pass

The batch reviewer checked all five modules in a single pass. Verdict: all PASS, no blocking issues. The reviewer noted several cross-module consistency items (Codex model base listed as "GPT-5.3 variant" instead of "o3 variant", MCP support table inaccuracies) which we fixed before committing. The source traceability lesson from Batch 1 paid off — no "researcher agent output" citations found.

Batch 3: Dependent Modules (04, 07, 08)

Ten researchers, three writers, one reviewer. Modules 04 and 07 passed review cleanly. Module 08 (Consumer AI Comparison) returned NEEDS REVISION with four blocking issues:

  1. Claude Memory contradiction — the most significant error in the entire curriculum. Module 08 listed Claude's cross-session memory as "None" in two tables and multiple prose sections. But Module 02 (written in Batch 2) documented Claude Memory as GA on all plans as of March 2026, and the Anthropic profile confirmed it. The Module 08 writer had based parts on an earlier understanding before the memory launch was integrated.
  2. Haiku 4.5 pricing error — listed at $0.80/$4 instead of $1.00/$5.00.
  3. Wrong free tier model — "GPT-4.1-mini" cited as ChatGPT's free model, but the profile says "GPT-5.2 Instant."
  4. Stale API pricing models — GPT-4.1 listed as mid-range when GPT-5.4 Mini is the current option.

The reviser fixed all four issues plus three non-blocking items (marked unverified community sources, noted Preview model status, clarified reasoning effort levels).

Lesson learned: Cross-module factual consistency is the hardest thing to get right when modules are written in parallel by different agents. Each agent has its own context window and may not have the latest information from sibling modules. The reviewer agent is essential for catching these contradictions — and the reviewer needs to read all completed modules, not just the one under review.

Batch 4: Capstone (Module 10)

Four researchers covered on-device AI, multimodal generation, enterprise governance, and open vs. closed ecosystems. The writer was interrupted mid-draft (idle with "interrupted" status) but recovered when prompted to continue. The final module (~3,800 words) synthesized themes across the entire curriculum and looked forward to Google I/O 2026, Microsoft Build, and WWDC.

What We Learned

What Worked

  • Parallel research is a multiplier. Launching 8-16 researcher agents simultaneously compressed what would be hours of sequential research into minutes. The structured return format (Key findings, Data points, Contradictions, Sources) made handoff to writers seamless.
  • The review-revise cycle catches real errors. Every module that went through individual review (Modules 00, 01, 08) had blocking issues found and fixed. Batch review caught fewer issues but still identified cross-module consistency problems.
  • Agent Teams enable coordination. The shared task list, direct messaging, and idle notifications made it possible to coordinate 5+ writers simultaneously. The team lead pattern (orchestrator + specialized workers) maps well to curriculum development.
  • Git commits at batch boundaries create natural checkpoints. We could have rolled back any batch without losing prior work.
  • Living documentation files (CURRICULUM.md, CURRENT_CYCLE.md) kept state consistent across compaction boundaries and agent switches.

What to Improve

  • Require primary URLs from day one. The Batch 1 source traceability issue wasted a full revision cycle. Future research agents should be instructed to return only claims with primary URLs — no intermediary citations.
  • Seed the glossary before writing begins. We added 8 terms in the synthesis pass that should have been defined before the writers started. Pre-seeding the glossary from the research phase would prevent definition drift.
  • Run the synthesizer after each batch, not just at the end. The ClawHub and MCP count discrepancies would have been caught earlier if we'd synthesized after Batch 2.
  • Writer agent reliability. The stalled-writer incident in Batch 2 was caused by message timing, not actual failures. A simple file-existence check before respawning would avoid unnecessary duplicate work.
  • Cross-module fact propagation. When Module 02 established that Claude Memory launched in March 2026, that fact needed to propagate to Module 08's writer. Future batches should include a "facts from completed modules" brief in each writer's prompt.

What's Next

The curriculum is complete but not frozen. It's designed as a living knowledge base. Key upcoming milestones:

  • Google I/O 2026 (May 19-20): Expect Gemini updates, Project Astra reveal, Android 17 AI features. Modules 00, 01, 03, 10 will need updates.
  • Microsoft Build 2026 (June 2-3): Copilot stack updates, Agent 365 GA details. Modules 04, 07, 08 affected.
  • WWDC 2026 (June 8-12): iOS 27, LLM Siri, Core AI framework. Module 10 will need significant expansion.
  • Derivative generation: HTML interactive guides, slide decks, ecosystem diagrams, comparison tables — all generated from module content.
  • Source verification: Several UNVERIFIED sources in SOURCES.md need primary confirmation.
