Phase 1: Foundation Layer

Before any component could be built, two foundational pieces had to exist: a localStorage abstraction and build pipeline integration for JavaScript files.

storage.js (7.6K)

The storage wrapper implements the full WS0 Section 2.5 API: namespaced key-value storage, schema versioning for future migrations, and graceful degradation when localStorage is unavailable (private browsing, storage full, disabled). Every component that persists state — quiz answers, progress checkmarks, dark mode preference, concept-gate completions — goes through this single abstraction rather than touching localStorage directly.

dark-mode.js (3.2K)

The dark mode toggle respects the system’s prefers-color-scheme setting as a default, then allows the user to override it. The preference persists via storage.js. The toggle button was added to the header of all three Handlebars templates.

Build Pipeline Integration

The build script (build.js) was updated to add jsPath to all three template contexts and copy JS files into the output directory in Layer 6. Templates received 11 <script> tags: storage.js and dark-mode.js load on all pages; the 9 component scripts load on module pages only. The inline keyboard shortcut script from Round 4’s module.hbs was preserved alongside the new navigation.js.

Build verification confirmed 14 HTML files output, 2 JS files copied to output, and correct relative paths across all templates.

The file:// CORS Problem

One of the key technical challenges in Round 5 was the file:// CORS constraint. The curriculum is designed to work offline — opened directly from the filesystem, no server required. But browsers enforce CORS restrictions on file:// URLs, which means fetch() calls to load JSON data files fail silently.

This affected glossary-tooltip.js directly. The tooltip component needs the full glossary dataset (55 terms with definitions, categories, and related terms) to render tooltips on hover. The original plan was to load this from a separate glossary.json file. That does not work on file://.

The solution: embed the glossary data as inline JSON inside a <script id="glossary-data" type="application/json"> tag in the template. The build pipeline serializes the glossary data at build time; the component reads it from the DOM at runtime. No fetch, no CORS. The same approach was used for the course map data (4 parts, 11 modules) that the progress tracker needs.

This was logged as a DECISION_LOG entry because it trades a small increase in HTML file size for guaranteed offline functionality — the right trade-off for this project’s delivery format.

The Annotation Injector Upgrade

Round 4’s annotation injector was a proof-of-concept: it could insert annotations into the right location in a module’s HTML, but it produced simplified markup that did not match the WS0 component library specification. A quiz annotation, for example, would render as a generic <div> with the quiz text, not as the structured markup that quiz.js expects to hydrate.

The upgrade was the critical path item for Round 5. The Quiz & Assessment engineer rebuilt the injector to produce WS0-compliant HTML for 5 component types: quiz (3 variants), self-explanation prompt, concept-check gate, worked-example fader, and generic annotation. Each component type has its own markup template with the correct CSS classes, ARIA attributes, and data attributes that the corresponding JavaScript component expects.

This matters because Round 6 will produce 11 YAML annotation files — one per module. Getting the injector right now means every annotation in Round 6 will produce correct, hydrate-ready markup from the start, rather than requiring a markup fix pass in Round 7.

Two test annotations (a concept-gate and a worked-example) were added to M00’s YAML file to verify the upgraded injector produces correct markup for component types beyond the 3 original proof-of-concept annotations from Round 4.

Bugs Found and Fixed

Session tracking selector too broad. The progress-tracker.js and storage.js session tracking used a CSS selector that matched all elements with an id attribute. When the glossary data was embedded as <script id="glossary-data">, the session tracker picked it up as a “section” of the page, skewing progress calculations. The fix: narrow the selector to h2[id], h3[id] only, matching actual content sections. Both storage.js and progress-tracker.js were patched.
verify.js class name mismatch. The verification script’s check #7 looked for Round 4’s simplified annotation class names (.annotation-quiz), but the WS0-compliant injector upgrade changed them to match the component library spec (.quiz, .self-explain). The check was updated to match the new class names. A minor issue, but it would have caused verify.js to report false failures if left uncorrected.

What We Learned

Vertical slices over horizontal layers. Round 4 organized agents by infrastructure layer (pipeline, CSS, templates). Round 5 organized by component domain (quiz/assessment, tables/tooltips, progress/navigation). The domain-based split was more effective for parallel work because each engineer could build their components end-to-end without waiting for another agent’s layer to be ready. The only sequential dependency was Phase 1 (foundation), which every component needed.
Inline data beats fetch for offline-first. The file:// CORS constraint forced a design decision early: embed data inline or require a local server. Inline JSON in <script type="application/json"> tags is the right pattern for this project. It adds a few KB to each HTML file but guarantees the curriculum works anywhere — USB drive, downloaded folder, air-gapped network. The trade-off is clearly in favor of portability.
Upgrade the injector before building components. Making the annotation injector upgrade the first task in Phase 2 was the right call. It meant every component engineer could test against real, spec-compliant markup from the start. If the injector had been upgraded last, the components would have been built against placeholder markup and then needed adjustment — or worse, the injector would have been shaped to match whatever markup the components happened to expect rather than the WS0 spec.
Broad CSS selectors cause subtle bugs. The session tracking bug (matching [id] instead of h2[id], h3[id]) was a reminder that selectors should be as specific as the intent. The selector was written before the glossary data script tag existed, and it was correct at the time. But “all elements with an id” is almost never what you actually mean. Narrowing selectors at the point of writing, not after a bug surfaces, is the better practice.
Verification checks must evolve with the code. When the annotation injector was upgraded to WS0-compliant class names, verify.js immediately started checking for the old names. The fix was trivial, but the pattern is important: every code change that affects testable output should also update the verification suite in the same work unit. Round 5 ended with 18 checks (11 from Round 4 + 7 new), and all 18 reflect the current state of the codebase.
