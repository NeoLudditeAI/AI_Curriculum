The Mission

Rounds 4 through 6 built text-based interactivity: a build pipeline, 11 JavaScript components, and 337 YAML annotations giving every module quizzes, self-explanations, worked examples, and more. But the curriculum was still entirely textual. Architecture relationships, process flows, competitive positioning, cost optimization strategies — all described in prose. Round 7’s job was to make them visual.

The scope: 21 HIGH-priority diagrams from the WS4 visual specifications, spanning all 11 modules. 15 Mermaid.js architecture diagrams, 3 interactive decision trees, 2 Chart.js data visualizations, and 1 recommendation quiz. All rendered client-side — no server, no build-time compilation, no external dependencies beyond vendored libraries. Every diagram works offline, respects dark mode, and degrades gracefully if JavaScript fails.

This was a hybrid engineering + content round. Unlike Round 6 (pure content authoring), Round 7 required building new infrastructure before any diagram could be authored. The critical path ran through a Pipeline Engineer who created 4 new JavaScript files, upgraded the annotation injector, vendored two libraries, and added 5 new verification checks — all before a single Mermaid diagram could be written.

Architecture Decisions

Three architectural decisions shaped Round 7, all logged in DECISION_LOG.md:

Client-side Mermaid.js over D2

The WS4 specifications originally called for D2, a standalone binary that compiles diagram source to SVG. But D2 requires installation in the build environment and a compilation step during builds. Mermaid.js renders directly in the browser from text source — no compiler, no build-time dependency, works offline, themeable via CSS custom properties. It handles all 15 static diagrams. The tradeoff: Mermaid’s visual output is less polished than D2’s, and some WS4 layouts needed adaptation. But for a static HTML curriculum that must work from file://, client-side rendering wins.

MutationObserver for dark mode

Round 5’s dark-mode.js toggles themes by setting data-theme on the document element, but it does not dispatch a custom event. Modifying it would violate the Round 5 boundary. New diagram components detect theme changes via MutationObserver watching the data-theme attribute — a standard browser API that requires zero changes to existing code. All four new JS files use this pattern consistently.

AFC_Storage for interactive persistence

Decision trees and the recommendation quiz need to remember user selections across sessions. Rather than using raw localStorage, all new components use window.AFC_Storage — the same persistence abstraction that Round 5’s quiz, concept-gate, and self-explanation components use. The Pipeline Engineer added decisionTrees and recommendations keys to the storage schema and bumped the version from 1 to 2.

Phase 1: Infrastructure

Phase 1 was a sequential gate — it had to complete before any content authoring could begin. The Pipeline Engineer built the entire diagram rendering stack:

  • 4 new JS files: mermaid-init.js (Mermaid configuration + dark mode observer), chart-renderer.js (Chart.js initialization + theme sync), decision-tree.js (interactive branching with keyboard navigation and AFC_Storage persistence), recommendation-quiz.js (weighted scoring form with platform recommendations)
  • 2 vendored libraries: mermaid.min.js (Mermaid v11) and chart.umd.min.js (Chart.js v4), copied from node_modules into build/js/ so Layer 6 of the build pipeline copies them to the output directory automatically
  • Annotation injector upgrade: Added diagram as a 6th component type with 4 variants (mermaid, chart, decision-tree, recommendation), each producing accessible <figure> markup with role="img" and aria-label
  • Template updates: 6 new <script> tags added to module.hbs for the vendored libraries and new component JS files
  • verify.js expansion: 5 new checks (19–23) covering diagram component type in injector, vendored library presence, YAML parsing, and diagram annotation field validation
  • storage.js update: decisionTrees: {} and recommendations: {} added to DEFAULT_STATE, schema version bumped from 1 to 2

After Phase 1, the Team Leader ran a baseline verification: node build/build.js produced 14 files with zero warnings, and node build/verify.js passed all 23 checks (18 existing + 5 new).

Diagram annotations were appended to the existing Round 6 YAML files. Each entry specifies an after section ID, the diagram component type, a variant, and a data block containing the diagram source (Mermaid text), configuration JSON (Chart.js/decision-tree), or scoring matrix (recommendation quiz). Every entry required an id, title, and alt field for accessibility.

Shared YAML coordination

Three modules had both a Mermaid Author and the Interactive Engineer writing to the same YAML file: M01, M02, and M04. The coordination protocol required the Mermaid Author to write first and signal completion, then the Interactive Engineer appended their decision tree entry. This prevented YAML corruption from simultaneous writes.

The Mermaid A Bottleneck

The most notable process observation from Round 7 was the workload imbalance between Mermaid Authors A and B. While Author B completed all 8 diagrams (M04, M05, M06, M10 + synthesis map) and was waiting idle, Author A was still working through M00 — the very first module.

The root cause: M00’s AI Ecosystem Map is one of the most complex diagrams in the curriculum. It maps four major ecosystems (Anthropic, OpenAI, Google, Microsoft) with their product hierarchies, competitive positioning, and specialist tool clusters. Translating the WS4 spec into valid Mermaid syntax while maintaining factual accuracy against the module text is fundamentally harder for dense ecosystem maps than for sequential process flows.

Author A was the critical path for the entire back half of the round. The Interactive Engineer’s M01 and M02 decision trees were blocked waiting for Author A to complete those modules’ Mermaid diagrams (due to shared YAML coordination). The CSS/QA accessibility verification was blocked waiting for all diagrams. The TL’s Phase 4 verification was blocked waiting for everything.

The bottleneck is a useful observation for future rounds: complex diagrams should be weighted more heavily in work allocation, and ecosystem-scale maps may warrant their own dedicated agent.

Interesting Challenges

Chart.js waterfall simulation

The M09-2 Cost Optimization Waterfall chart required a waterfall-style visualization showing how token costs decrease through successive optimization strategies (caching, batching, model tiering). Chart.js has no native waterfall chart type. The Interactive Engineer simulated it using a stacked bar chart with transparent base segments — each bar floats above the previous one’s endpoint, creating the waterfall effect. The trick is calculating the invisible “base” dataset that offsets each visible bar to the correct starting position.

Four diagram variants, one component type

The annotation injector upgrade was conceptually straightforward but architecturally interesting. Rather than adding 4 new component types (which would have complicated the YAML schema and the injector’s dispatch logic), the Pipeline Engineer added a single diagram component type with a variant field that dispatches to 4 different render functions. This mirrors how Round 5’s quiz component uses variants (concept-check, section-review, module-assessment) — a pattern that proved its worth in Round 6 and scales naturally here.

Mermaid dark mode re-rendering

Mermaid.js renders diagrams to SVG at initialization time, baking in the color scheme. When the user toggles dark mode, the existing SVGs become unreadable (dark text on dark background or vice versa). The solution: mermaid-init.js detects theme changes via MutationObserver, re-initializes Mermaid with updated theme colors, and forces a full re-render of all .mermaid elements. This is heavier than CSS-only theming but necessary because Mermaid’s SVG output uses inline styles that CSS custom properties cannot override.

What We Learned

Hybrid rounds need sequential gates. Round 6 was embarrassingly parallel — 4 authors, no dependencies. Round 7 had a hard sequential dependency: Phase 1 infrastructure had to complete before any diagram could be authored. This is the correct architecture (content authors need working rendering infrastructure), but it means the Pipeline Engineer is a single point of failure for the entire round. Future hybrid rounds should keep Phase 1 as lean as possible and test thoroughly before unblocking content agents.
Diagram complexity varies wildly. A simple process flow (M03 Gather-Act-Verify Loop) might take an agent 10 minutes. An ecosystem-scale map (M00 AI Ecosystem Map) with dozens of nodes, multiple subgraphs, and cross-cutting relationships can take 10x longer. Work allocation should weight diagram complexity, not just diagram count. Mermaid Author B completed 8 diagrams faster than Author A completed 7 because the M04–M10 diagrams were structurally simpler.
Shared YAML files create coordination bottlenecks. The protocol requiring Mermaid Authors to write before the Interactive Engineer could append worked correctly — no YAML corruption occurred. But it meant the Interactive Engineer sat idle waiting for Author A’s M01 and M02 completions while Author B had long since finished M04. The Interactive Engineer used this time productively (completing M08 charts and the recommendation quiz first), but the dependency chain still lengthened the critical path.
Client-side rendering trades polish for portability. Mermaid’s browser-rendered SVGs are less visually refined than D2’s compiled output. Labels can overflow, long text wraps unpredictably, and complex subgraph layouts sometimes produce suboptimal spacing. But the diagrams render instantly, work offline, support dark mode via re-initialization, and require zero build infrastructure. For a curriculum designed to be opened as local HTML files, this tradeoff is correct.
The variant pattern scales. Round 5 established the pattern: one component type with multiple variants dispatched in the injector. Round 6 validated it at scale (337 annotations across 5 types). Round 7 extended it (diagram type with 4 variants). The pattern keeps the YAML schema stable, the injector logic centralized, and the component JS files focused. It is now a proven architectural pattern for this project.
