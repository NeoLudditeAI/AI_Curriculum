The Mission

Round 3 produced 81,400 words of pedagogical design specifications across 9 workstream documents: component library, course architecture, module-level designs, assessment framework, visual specs, engagement design, layout and accessibility, and spaced repetition. All of it was theory on paper. Round 4’s job was to make it real — build the foundation that turns 11 markdown modules into a navigable, styled, accessible HTML curriculum.

The scope was deliberately constrained. Round 4 builds the pipeline, the design system, and the templates. It does not build the 9 interactive JavaScript components (Round 5), the full annotation content (Round 6), or the final polished export (Round 7). The goal was a working proof-of-concept: take Module 00 (Landscape Overview) from markdown to fully rendered HTML with annotations injected, glossary terms linked, and navigation working — then verify it visually in Chrome at three breakpoints.

The Team

Round 4 used a focused 6-person team, the smallest so far. Where Round 1 needed 38+ researcher agents and Round 3 needed 10 pedagogical designers, Round 4 needed engineers.

Phase 0: Content Preparation

Before the pipeline could be built, three housekeeping tasks needed to happen. The Content Prep agent handled all of them.

M00 annotation skeleton. The proof-of-concept needed real annotations to inject. The agent created a YAML annotation file for Module 00 with 3 annotations, each targeting verified section IDs in the markdown. This gave the pipeline something concrete to work with without requiring the full 11-module annotation pass that Round 6 will handle.

M07 table patch. The pedagogical design called for a synthesized scheduled-tasks comparison table in Module 07 (Skills, Plugins & Automation). This was a content gap identified in Round 3 — the module discussed scheduled tasks across platforms but lacked a unified comparison. The table was inserted directly into the module markdown.

SOURCES.md housekeeping. Seven issues were resolved: 3 URLs returning 404 were archived with notation, 3 duplicate source entries were consolidated, and 1 orphaned source (cited in a research brief but never referenced in any module body) was removed. These had been carried as known issues since Round 2.

Phase 1: Build Pipeline

The Pipeline Engineer built the core transformation system: a 6-layer unified/remark pipeline that takes markdown modules and produces styled HTML. The pipeline processes all 11 modules, injecting annotations from YAML files, detecting and linking glossary terms, rewriting cross-module links from .md to .html, and generating navigation data.

Key implementation details:

  • 3 custom remark plugins for annotation injection, glossary term detection, and cross-module link rewriting
  • A data generator that parses GLOSSARY.md into structured JSON for the glossary page
  • 55 glossary terms detected across the corpus, with 20 first-use instances identified in M00 alone
  • Idempotent builds: consecutive runs produce identical output (verified via MD5 comparison)
  • 112 npm packages installed, the first external dependencies in this project
  • Scaffolding Handlebars templates created as a starting point for Phase 3 enhancement

The pipeline produced 14 output files on its first successful run: 11 module HTML pages, an index/course map page, a glossary page, and a build manifest.

Phase 2: CSS Design System

The Design System Engineer translated the WS6 layout and accessibility specification (4,485 words of design requirements) into 7 CSS files totaling 1,101 lines. The system implements the full WS6 spec:

  • Light and dark color modes using CSS custom properties, with automatic detection via prefers-color-scheme
  • Type scale matching WS6 Section 6.1: Inter for body text, JetBrains Mono for code, imported via Google Fonts
  • 3-column responsive layout that collapses gracefully across breakpoints
  • Print stylesheet for clean paper output
  • Focus management with visible focus rings for keyboard navigation
  • Reduced motion support via prefers-reduced-motion media query
  • WCAG 2.2 AA compliant color contrast ratios

Phase 3: Page Templates and Navigation

The Template Engineer enhanced the scaffolding templates from Phase 1 into full WS6-compliant Handlebars templates. Three templates were built:

module.hbs

The module template is the most complex. It includes skip-to-content links, ARIA landmark roles (banner, navigation, main, complementary, contentinfo), breadcrumb navigation, a sidebar table of contents that highlights the current section, and previous/next module navigation with keyboard shortcuts (left and right arrow keys). Round 5 component placeholders are present as data-* attributes, ready for JavaScript hydration.

index.hbs

The course map groups all 11 modules into the 4-Part learning sequence defined in WS1: The Terrain (M00–M02), Agent Systems (M03, M06, M04, M05), Building & Automating (M07, M09), and Synthesis & Horizon (M08, M10). Each Part has its own accent color. A resume-where-you-left-off banner placeholder is included for Round 5’s progress tracker component.

glossary.hbs

The glossary template renders all 55 terms with a real-time filter input. The filter uses aria-live to announce result counts to screen readers. Terms are organized using <dl> semantics with category grouping. Related terms link to each other via resolved slugs.

The navigation system follows the non-numerical learning sequence throughout: M00, M01, M02, M03, M06, M04, M05, M07, M09, M08, M10. The “Next Module” link on M03 points to M06, not M04. This was verified across all 11 module pages.

Phase 4: Integration and Verification

The final phase brought everything together and verified it worked. The Pipeline Engineer created verify.js, an automated verification script with 11 checks:

  • All 14 expected output files exist
  • Every HTML file is valid (no unclosed tags, correct DOCTYPE)
  • M00 contains all 3 injected annotations
  • Glossary terms are linked in module content
  • Cross-module links resolve correctly (.md rewritten to .html)
  • Navigation sequence matches the non-numerical learning order
  • CSS files are referenced and loadable
  • ARIA landmarks are present in every page
  • Build is idempotent (two consecutive runs produce byte-identical output)
  • No JavaScript console errors in any generated file
  • All internal links resolve to existing files

All 11 checks passed. Chrome visual verification was then performed at three breakpoints: 1280px (desktop), 768px (tablet), and 375px (mobile). Zero console errors across all pages.

Bugs Found and Fixed

Integration exposed several issues that were caught and resolved during Phase 4:

M01 duplicate heading ID. Module 01 (Models & Intelligence) had two sections that generated the same HTML ID, causing the sidebar TOC to link to the wrong section. The duplicate heading was renamed to produce a unique slug.
Cross-module link rewriting. The initial pipeline correctly rewrote .md extensions to .html in cross-module links, but did not handle relative paths with directory prefixes. Links like ../modules/MODULE-03.md needed the path prefix stripped since all output files live in the same directory. The remark plugin was updated to normalize paths before rewriting.
Glossary related-term slug resolution. The glossary template linked related terms to each other, but some slugs were generated from the term display name rather than the canonical GLOSSARY.md identifier. This caused broken links for terms with special characters or abbreviations. The data generator was updated to use a consistent slugification algorithm.
Mobile overflow. At the 375px breakpoint, comparison tables overflowed their container on modules with wide tables (particularly M08 and M09). A horizontal scroll wrapper was added to the table CSS, with a subtle shadow indicator when content is scrollable.

What We Learned

Small, focused teams ship faster. Round 1 used 38+ agents. Round 3 used 10. Round 4 used 5 (plus a chronicler). The smaller team had clearer ownership boundaries and less coordination overhead. Each agent owned a distinct deliverable with minimal overlap. The constraint was not capacity but dependency ordering — Phases 2 and 3 could not start until Phase 1 delivered the scaffolding templates and build system.
Phase gating prevents compounding errors. The 5-phase structure (content prep, pipeline, CSS, templates, verification) meant each layer was solid before the next began. When Phase 4 found the mobile overflow bug, it was clearly a CSS issue, not a template issue or a pipeline issue. Clean layer boundaries made debugging straightforward.
Proof-of-concept scope was right. Building the full pipeline for all 11 modules but only annotating M00 was the correct trade-off. It proved the pipeline works end-to-end without requiring 11 YAML annotation files to be authored first. The scaffolding is ready; Round 6 fills in the content.
SOURCES.md housekeeping was overdue. The 7 issues resolved in Phase 0 had been flagged since Round 2 and carried across two full rounds. Dedicating a phase to clearing technical debt before building new infrastructure was the right call — it prevented downstream confusion about source integrity.

What’s Next

Round 4 delivered the foundation. The pipeline works, the pages render, the navigation is correct, accessibility landmarks are in place, and M00 proves the annotation injection system functions. Three rounds remain:

  • Round 5 (Component Build): 9 interactive JavaScript components — quiz widget, accordion, table enhancer, glossary tooltip, concept-check gate, worked-example fader, self-explanation prompt, progress tracker, and navigation controls. These hydrate from the data-* attribute placeholders already embedded in the templates. localStorage integration for progress persistence. Dark mode toggle UI.
  • Round 6 (Content Production): 11 YAML annotation files (one per module), 40 diagram source files (21 high priority), capstone expert answer document, curiosity hooks, spiral callbacks, opening hooks, and closing synthesis exercises — all specified in the Round 3 workstream documents.
  • Round 7 (Assembly & QA): Full 11-module build with all annotations, cross-browser testing, WCAG 2.2 AA accessibility audit, performance optimization, print stylesheet verification, and the final single-HTML curriculum export.

The foundation is laid. Now we build on it.
