# Part 2 Capstone Expert Answer: The Agent Design

**Scenario:** Build a customer support agent: classifies urgency, searches a 50K-document knowledge base, drafts responses, escalates complex cases. 200 tickets/day, $500/month API budget.

---

## Expert Answer

**Single vs. Multi-Agent: Multi-agent with hierarchical delegation.** A single agent could handle this, but the workload divides naturally into distinct roles with different compute profiles. A router agent classifies urgency (simple classification -- cheapest model). A researcher agent searches the knowledge base (retrieval -- fast model). A drafter agent composes responses (generation -- mid-tier model). An escalation agent identifies cases exceeding automated resolution (judgment -- mid-tier to flagship). Running these as separate agents enables parallelism (process multiple tickets simultaneously) and specialization (each agent optimized for its task). At 200 tickets/day, the cost savings from model tiering across roles more than justify the orchestration overhead.

**Orchestration Pattern: Router with fan-out.** The router agent classifies each incoming ticket by urgency (critical/high/normal/low) and type (billing/technical/account/other), then dispatches to the appropriate handler. For normal tickets, the researcher and drafter run sequentially (pipeline pattern: search, then draft). For critical tickets, the escalation agent receives the ticket in parallel with the drafter, so a human is notified immediately while a draft response is still prepared.

**MCP Servers Needed:** Three. (1) A knowledge base MCP server exposing semantic search over the 50K documents via the tools primitive. (2) A ticketing system MCP server (Zendesk, Freshdesk, or custom) for reading tickets and posting responses. (3) A notification MCP server (Slack or email) for escalation alerts. MCP's standardized protocol (Module 06) means these servers work with any MCP-compatible agent framework.

**Model Selection Per Role:** Router/classifier: Haiku 4.5 ($1/$5 per MTok) -- simple classification where speed matters most. Researcher: Haiku 4.5 -- retrieval and extraction from search results. Drafter: Sonnet 4.6 ($3/$15 per MTok) -- needs coherent, professional tone. Escalation judge: Sonnet 4.6 -- needs nuanced judgment about complexity. At 200 tickets/day with ~3K tokens per ticket cycle, estimated monthly cost: ~$350-$450, within the $500 budget. The router and researcher use the cheapest tier (Module 01 model selection framework); the drafter and escalation judge use mid-tier where quality directly affects customer experience.

**Safety Model:** Human approval required for: any response to critical-urgency tickets, any ticket involving account changes or billing disputes, and any response the drafter flags as low-confidence. Pre-approved: standard responses to common questions (how-to, status checks, documentation pointers). Default-deny on knowledge base write access -- agents can read but never modify the 50K documents. Guardrails: input validation rejects tickets containing potential prompt injection; output validation checks drafts against a prohibited-content list before sending. All agent actions logged to an immutable audit trail for compliance review.

**Deployment Platform: Cloud vs. Self-Hosted (OpenClaw).** Should this agent run on a cloud platform (Claude Agent SDK, OpenAI Agents SDK) or self-hosted via OpenClaw? The answer depends on the organization's constraints. A cloud deployment on Anthropic's infrastructure is operationally simpler -- no servers to manage, built-in scaling for the 200 tickets/day volume, and the Agent SDK handles orchestration natively. But OpenClaw offers a compelling alternative for organizations with strict data residency requirements or cost sensitivity. OpenClaw's local-first architecture means every ticket, knowledge base query, and drafted response stays on-premises -- critical for support operations handling regulated data (healthcare, finance, government). The Docker deployment provides isolation between the agent runtime and the host system, and LLM provider flexibility means you can route to Anthropic's API for quality-critical drafting while using Ollama with an open-weight model for simple classification tasks, reducing API costs by 70-90% on the routing layer.

The tradeoff is operational burden. With OpenClaw, the team must manage the Gateway, configure channels (Slack, email, web widget), and maintain the Heartbeat system for any background monitoring tasks. ClawHub skills could replace some custom MCP server development -- for example, a pre-built Zendesk skill from ClawHub instead of a custom ticketing MCP server -- but ClawHub's 12-20% malicious skill rate (Module 05) means every skill must be audited before deployment. For a support agent handling customer data, this security risk is non-trivial. The recommended approach: use ClawHub skills only from verified publishers, pin to specific versions, and run the entire stack in Docker with network restrictions. For most teams without dedicated DevOps capacity, the cloud platform's managed infrastructure and built-in safety controls justify the higher per-token cost. For organizations with an existing container orchestration stack and data sovereignty mandates, OpenClaw's self-hosted model delivers the same multi-agent architecture at substantially lower operating cost.
